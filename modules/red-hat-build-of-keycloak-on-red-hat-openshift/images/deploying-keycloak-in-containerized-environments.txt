In modern architecture, the Red Hat build of Keycloak is designed to run as an immutable container. Containerization allows for consistent deployment across environments, ranging from your local development machine to large-scale production clusters.

When deploying the Red Hat build of Keycloak in a containerized environment, you should adhere to three core principles. First, focus on immutability by treating your containers as ephemeral; instead of modifying a running container, you should always rebuild and redeploy the image with updated configurations or versions. Second, prioritize performance optimization by using optimized images to ensure fast startup times, which is essential for auto-scaling. And third, emphasize security isolation by leveraging container-native features to protect sensitive credentials and network traffic.

For environments where rapid startup and pre-optimized image layers are not your primary constraint, you can deploy the standard Red Hat build of Keycloak image. To run this, you would use a command similar to the `podman run` command shown in your materials, which maps the ports and sets your administrative environment variables using the `start-dev` flag.

Regarding initial administrator credentials, it is important to know that by default, the Red Hat build of Keycloak restricts the creation of the initial administrator to local network connections. Because containers create their own network boundary, you must explicitly provide administrative credentials via environment variables during the initial container startup. You will use `KEYCLOAK_ADMIN` to set the username and `KEYCLOAK_ADMIN_PASSWORD` to set the initial password. 

An important caution here: always manage these credentials using secure mechanisms such as OpenShift Secrets or Kubernetes Secrets. Avoid hardcoding them directly into command-line scripts or container manifest files.
