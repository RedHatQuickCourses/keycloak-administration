In this lab, you will learn how to configure Red Hat build of Keycloak to act as an Identity Broker. By integrating an external Identity Provider, you will allow users to authenticate using credentials from an external source and map that incoming identity information directly to your local realm.

Before we begin, ensure you have a running instance of Red Hat build of Keycloak, administrative access to the Admin Console, and access to an external Identity Provider—such as a Google developer account or a SAML-based provider.

By the end of this session, you will be able to configure a new Identity Provider, set up mappers to sync user attributes, and successfully test the authentication flow.

Let's start with Exercise 1: Configuring an Identity Provider. This creates the trust relationship between your environment and the external provider. First, log in to the Keycloak Admin Console. From the left-hand menu, click on "Identity providers." Select your desired provider from the "Add provider" dropdown, such as Google or SAML v2.0. You will then need to configure the required fields based on your external provider's specific documentation. Pay close attention to the Redirect URI; you must copy this value and paste it into your external provider's configuration console as the Authorized Redirect URI. Also, enter the Client ID and Client Secret you obtained from the external provider, and then click Save.

Next, in Exercise 2, we will implement Identity Provider Mappers. These mappers are essential because they allow you to propagate information from the external provider into your local user sessions. Within the configuration for the Identity Provider you just created, navigate to the Mappers tab and click "Add mapper." To extract identity information, you will set the Name to "Sync Email," choose the "Attribute Importer" as your Mapper type, and then set both the Identity Provider Attribute and the User Attribute to "email." Once configured, click Save. 

As a quick note, if you are using Identity Brokering APIs V2, keep in mind that Keycloak can store tokens issued by the external provider. If your application needs to call the external provider's API on behalf of the user, ensure that the "Store tokens" option is enabled in the provider settings.

Moving on to Exercise 3, we will verify the integration. Open your application or the Keycloak Account Console in an incognito browser window and navigate to the login page. You should now see a button for your external Identity Provider. Click that button and authenticate using your external credentials. Once you have authenticated, check the Users section in the Admin Console. You should see a new user account either created or successfully linked, with the attributes correctly mapped from your external provider.

If you encounter any issues, here are a few troubleshooting tips. If you receive a "Redirect URI Mismatch" error, double-check that the URI in Keycloak matches exactly what is configured in the external provider's portal. If you are using SAML and encounter protocol errors, verify that the metadata URL or XML file is reachable by your Keycloak instance. For deeper analysis, you can monitor the server logs to debug authentication handshakes by running the command: `tail -f /opt/keycloak/data/log/keycloak.log`.

To summarize, you have successfully configured Red Hat build of Keycloak as an Identity Broker, mapped external attributes to your local database, and verified the authentication flow. Implementing this architecture allows for a centralized user management strategy across different security domains.
