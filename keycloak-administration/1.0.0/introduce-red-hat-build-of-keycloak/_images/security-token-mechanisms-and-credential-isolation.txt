In distributed architectures, managing identity securely across disparate trust domains requires robust token mechanisms. Red Hat build of Keycloak leverages industry-standard protocols to ensure that credentials remain isolated and that access is strictly controlled, even when services operate under different security authorities.

Let's begin by discussing how these token mechanisms maintain credential integrity. At the core of the Keycloak security model is the issuance and validation of tokens. To prevent the misuse of credentials, Keycloak adheres to the principle of credential isolation. This ensures that a service receiving a token cannot use that same token to impersonate the user or client against a different, unauthorized service.

A critical mechanism for preventing token theft and replay attacks is known as Proof of Possession, or PoP. In this workflow, when a client requests a token, it cryptographically signs the request using its private key. The resource server then receives the token, which contains a fingerprint of the client's public key. The server validates the request by checking the client's signature against this embedded fingerprint. The result is that even if an attacker intercepts the token, they cannot use it because they lack the private key required to sign the requests, effectively binding the token to the legitimate owner.

When applications must access resources distributed across multiple trust domains, each with its own authorization server, Keycloak supports OAuth Identity and Authorization Chaining. This approach removes the need to share long-lived user credentials across domains.

This process, which utilizes Token Exchange and the JWT Profile for Client Authentication, follows a specific flow:
First, the client in the first trust domain identifies the authorization server of the second domain.
Next, the client presents its current token to the first authorization server, which returns a JWT authorization grant designed specifically for the second domain.
The client then presents this JWT grant to the second authorization server. Once validated, that server issues a localized access token. Finally, the client uses this new token to interact with the protected resource in the second domain.

It is important to remember that this mechanism ensures the access token granted by the second domain has limited privileges. It is scoped specifically to the resources within that domain. This ensures that if the second domain were compromised, the stolen token could not be leveraged to access resources in the first domain or other external services.

Now, let’s move on to a hands-on activity where you will observe how tokens are restricted to specific audiences to ensure isolation.

First, access the Admin Console by logging into your Keycloak instance and navigating to the Clients section.
Second, select a client and navigate to the Client Scopes tab. Here, take note of how specific claims are attached to the token.
Third, test the token request. Using the `kcadm.sh` CLI or a REST client, request a token for the client. You can use the command: `./kcadm.sh get tokens --realm myrealm --client myclient --user myuser`.
Fourth, decode and verify your results. Use a tool like `jwt.io` to decode the generated access token and observe the `aud`, or audience claim. Ask yourself: Is the audience restricted to the intended resource server, or is it overly permissive?
Finally, review the credential isolation. You will observe that the token contains no raw user credentials, such as passwords. This isolation ensures that even if the token is logged or intercepted at the resource server, the user's master credentials remain entirely shielded within the Keycloak identity provider.
