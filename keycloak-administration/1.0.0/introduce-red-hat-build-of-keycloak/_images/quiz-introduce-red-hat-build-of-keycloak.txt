Now it is time to test your knowledge of the fundamental concepts and architectural pillars of the Red Hat build of Keycloak. Let’s walk through a few questions to reinforce what you have learned.

First, let's look at identity protocols. Which of the following are natively supported by the Red Hat build of Keycloak for identity and access management? The correct answers are OpenID Connect, commonly known as OIDC; OAuth 2.0; and SAML 2.0. This is because the Red Hat build of Keycloak is specifically designed to function as an identity provider that supports these industry-standard protocols. By supporting OIDC—which is built on top of OAuth 2.0—and SAML 2.0, the platform allows for seamless integration with both modern applications and legacy systems.

Next, consider the architectural design regarding credential isolation. What is its primary purpose? The correct answer is to ensure that the identity provider manages credentials, keeping them isolated from individual application databases. This approach to credential isolation is a core security benefit. By centralizing authentication, your applications never need to handle raw user credentials. This significantly reduces the attack surface and simplifies your compliance requirements.

Moving on to performance, is it true or false that the Red Hat build of Keycloak 26.6 provides configuration options to optimize for faster startup times and reduced memory footprints? The answer is true. As detailed in the configuration documentation, administrators can apply specific build options and configuration methods to tailor the runtime environment for performance, ensuring the service is perfectly optimized for the specific requirements of your host environment.

Finally, which concept best describes the role of the Red Hat build of Keycloak in a Single Sign-On, or SSO, environment? It acts as a central identity provider that manages authentication and authorization across multiple disparate applications. By operating as a central identity provider, Keycloak effectively decouples the authentication logic from the application itself. This allows your users to authenticate just once and gain access to multiple authorized services without the need to re-enter their credentials.
