In this guided exercise, we will walk through the post-deployment configuration of the Red Hat build of Keycloak on an OpenShift cluster. Our goal is to ensure that after deploying the operator and the Keycloak custom resource, your environment is properly optimized for secure, production-like operations.

Before we begin, please ensure you have a running Red Hat OpenShift cluster, the Red Hat build of Keycloak Operator installed via OperatorHub, and an existing Keycloak custom resource instance already deployed in your project.

Let's start with the first task: verifying the health of your Keycloak custom resource. Before modifying any configurations, we must confirm the instance is ready. First, log in to your OpenShift cluster using the command line interface with `oc login -u <username>`, and then switch to your project using `oc project <your-keycloak-project>`. Once you are in the correct project, verify the status of your instance by running `oc get keycloak <instance-name> -o yaml`. You are looking to confirm that the `status.ready` field is set to `true`.

Next, we will move on to Task 2: configuring TLS and Ingress. In OpenShift, the Keycloak route is managed by the Ingress Controller, and ensuring proper TLS termination is critical for security. Start by retrieving the default route created by the operator using `oc get routes`. If you are using custom certificates, you can patch the route to point to the secret containing your certificate and key. Use the `oc patch route` command with a merge type to update the specification for TLS termination, as shown in the example provided in the lab materials.

For Task 3, we will update the environment variables. You can modify the configuration of your running instance by editing the Keycloak custom resource, which will trigger a rolling update of the pods. Run `oc edit keycloak <instance-name>` to open the resource configuration. Within this file, you can add or modify the `spec.additionalOptions` section to include settings like custom build options, database connection pooling, or logging levels. For instance, you might set the `http-relative-path` to `/auth` or adjust the `log-level` to `INFO`. Once you save and exit the editor, the operator will automatically reconcile the pods to reflect your changes.

Moving to Task 4, let's validate those configuration changes. After the pods have had time to restart, you can monitor the rollout status by running `oc rollout status statefulset/<instance-name>`. To be absolutely certain the configuration was applied, you can inspect the environment variables directly inside the pod using the command `oc set env pod/<instance-name>-0 --list | grep LOG_LEVEL`.

If you encounter any issues, refer to our troubleshooting section. If you experience a `Pod CrashLoopBackOff`, check the logs of the Keycloak operator to identify invalid configuration parameters in the `additionalOptions` by running `oc logs deployment/keycloak-operator-controller-manager -n <operator-namespace>`. If you find that a route is missing, verify that the `hostname` field in your custom resource is correctly specified and that the operator has the necessary permissions to create OpenShift routes.

A final tip for your production environments: always refer to the official documentation for Configuring Red Hat build of Keycloak to ensure that your chosen configuration options are fully supported in the OpenShift containerized distribution.
