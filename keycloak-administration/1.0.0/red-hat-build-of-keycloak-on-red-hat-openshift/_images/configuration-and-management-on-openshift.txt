When deploying the Red Hat build of Keycloak on OpenShift, it is important to understand that configuration and management shift from traditional file-based editing to a declarative, container-native approach. In this environment, you will rely on the OpenShift Operator for lifecycle management, meaning your key configurations—such as environment variables, secrets, and truststores—are managed through custom resources that the operator reconciles.

One critical aspect of this setup is managing your truststore. To secure communication between Keycloak and external services, such as an OpenShift API server for identity brokering, you must use a custom truststore. Please note that the certificate of your OpenShift 4 instance must be stored within this truststore, and you must explicitly configure your Keycloak server to utilize it.

To integrate an OpenShift version 4 instance as an identity provider, start by retrieving your API URL. You can do this by running the command `oc cluster-info`. From the output, look for the line that says "Kubernetes master is running at," followed by your API endpoint URL. Once you have that, navigate to the Keycloak Admin Console, select "Identity Providers" from the menu, and choose "OpenShift v4" from the "Add provider" dropdown. From there, input your Client ID, Client Secret, and the Base URL you just retrieved.

When planning for production environments that require high availability, you must ensure your components align with supported versions. Specifically, you will need at least OpenShift Container Platform version 4.17 or higher. If you are performing multi-cluster deployments on Red Hat OpenShift Service on AWS, or ROSA, you must also use Red Hat Data Grid version 8.5.x and AWS Aurora PostgreSQL version 17.x.

A very important warning: the image-based configuration provided by the Red Hat build of Keycloak is designed exclusively for OpenShift. Using these specific configurations on generic Kubernetes distributions is not supported.

Now, let’s walk through a guided exercise on configuring identity brokering. Before beginning, ensure you have a running instance of the Red Hat build of Keycloak on OpenShift and that the cluster certificate has already been imported into your truststore.

First, access your Admin Console using your administrator credentials. Second, navigate to "Identity Providers" and select "Openshift v4" from the "Add provider" list. Third, define your parameters: enter the Client ID and Secret generated for your client within the OpenShift project, and input the Base URL you retrieved earlier using the `oc cluster-info` command. Make sure to copy the Redirect URI provided by the console, as you will need this to finish configuring the OAuth client in your OpenShift console. Finally, save your configuration. Once finished, users will see the "Log in with OpenShift" option available on your login page.

If you encounter any issues, here are two troubleshooting tips. If authentication fails with a "PKIX path building failed" error, verify that your OpenShift API server certificate has been correctly injected into the Keycloak Truststore secret and is properly referenced in the Keycloak custom resource. Additionally, always double-check that your OpenShift version meets the minimum requirement of 4.17 to ensure full support for all Keycloak Operator features.
